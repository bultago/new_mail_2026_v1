## 위험 대응 전략

### R-01: 대규모 JSP 수정
**대응 계획**:
1. JSP 태그 자동 변환 스크립트 개발
2. 모듈별 순차 전환 (test → 독립 → 보조 → 핵심)
3. 각 모듈 전환 후 즉시 테스트

**모니터링**:
- 전환 진행률 주간 추적
- 에러율 모니터링

### R-02: 성능 저하
**대응 계획**:
1. Phase 1에서 성능 기준선 측정
2. 각 Phase 완료 시 성능 테스트
3. 성능 저하 발견 시 즉시 튜닝

**성능 목표**:
- 로그인: < 400ms (현재 ~500ms)
- 메일 목록: < 800ms (현재 ~1000ms)
- 메일 읽기: < 600ms (현재 ~800ms)

### R-03: 보안 이슈
**대응 계획**:
1. Phase 5에서 OWASP Top 10 검증
2. 보안 스캔 도구 실행 (OWASP ZAP, Burp Suite)
3. 외부 침투 테스트 실시

### R-04: 일정 지연
**대응 계획**:
1. 각 Phase에 버퍼 기간 15% 확보
2. 주간 진행률 모니터링
3. 지연 발생 시 리소스 추가 투입

**버퍼 기간**:
- Phase 1: 2주
- Phase 2: 3주
- Phase 3: 3주
- Phase 4: 4주
- Phase 5: 2주

### R-05: 데이터 손실
**대응 계획**:
1. 각 Phase 시작 전 백업 생성
2. 중요 작업 전 추가 백업
3. 롤백 테스트 정기 실행

**백업 전략**:
- 일일 백업: 개발 환경
- 주간 백업: 테스트 환경
- 월간 백업: 프로덕션 환경

## 위험 모니터링

### 주간 위험 검토 회의
- **시간**: 매주 월요일 오전 10시
- **참석자**: PM, 기술 리더, 개발 리더, QA 리더
- **안건**:
  - 지난 주 발생 위험 검토
  - 현재 위험 상태 평가
  - 새로운 위험 식별
  - 대응 조치 효과 확인

### 위험 에스컬레이션
```
발생 → 팀 리더 보고 → 대응 조치
  ↓ (해결 불가)
PM 에스컬레이션 → 경영진 보고
```

## 롤백 계획

### Phase별 롤백 포인트
```
Phase 2 완료 후: Spring 2.5로 롤백 가능
Phase 3 완료 후: iBATIS로 롤백 가능
```

4. 이전 버전 배포
5. 검증 테스트
6. 원인 분석 및 재발 방지
